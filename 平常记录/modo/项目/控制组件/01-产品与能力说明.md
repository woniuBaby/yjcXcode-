# 控制组件：产品与能力说明

## 1. 背景

业务希望接入 iOS 系统中的“控制”能力，让用户可以在控制中心、锁屏、操作按钮等系统入口放置一个按钮。用户点击后进入游戏，并在条件满足时自动收取已成熟鲜花。

目标渠道：

- 服务器：国内
- 渠道：iOS
- 系统能力：iOS 18 Control Widget

## 2. 目标

控制组件能力面向游戏侧提供一次“来自 iOS 控制入口的启动意图”，用于驱动自动收花流程。

目标行为：

- 游戏处于后台时，用户点击控制入口后拉起游戏。
- 游戏未运行或未登录时，用户点击控制入口后打开游戏，游戏按现有流程自动登录。
- 游戏登录和主场景 ready 后，如果存在可采摘鲜花，则自动完成收花。
- 如果不存在可采摘鲜花，则只进入游戏，不执行额外操作。
- 如果控制拉起后用户主动切后台或被系统事件打断，且收花接口尚未开始执行，则取消本次自动收花。

## 3. 非目标

- 不通过模拟点击、坐标点击、无障碍能力或私有 API 完成收花。
- 不在 Widget Extension 内直接执行完整游戏业务。
- 不要求服务端绝对证明“请求一定来自系统控制按钮”。服务端只把 `source=ios_control` 作为触发来源与埋点字段，实际业务仍以登录态、用户状态、鲜花成熟状态和幂等校验为准。
- 不支持用户被打断后很久再手动打开 App 仍继续执行旧的自动收花请求，除非产品后续明确要求。

## 4. 能力边界

Control Widget 的职责是提供系统入口和点击动作。它可以通过 AppIntent 写入一次性请求，并要求系统拉起主 App。

主 App / NativeBility 的职责是识别这次请求是否有效，尽早消费 App Group 标志，转成内存态任务，并在游戏层 ready 后返回给游戏侧。

游戏层 / 服务端的职责是判断是否存在可收取鲜花，并执行实际收花。

推荐职责划分：

| 模块 | 职责 |
|------|------|
| ControlWidget | 展示系统控制按钮 |
| AppIntent | 用户点击后写入 App Group pending request，并拉起主 App |
| App Group | Control Extension 与主 App 的短时共享通道 |
| NativeBility 控制插件 | `query` / `getInfo` / `delegate`：读 App Group `ControlWidget`、TTL 校验、读后消费 |
| 游戏层 | 等登录、弹窗、场景 ready 后调用 NativeBility 获取请求，并决定是否触发自动收花 |
| 服务端 | 校验登录态、requestId 幂等、成熟状态，执行或拒绝自动收花 |

## 5. 用户流程

### 5.1 正常自动收花

```text
用户点击系统控制按钮
  ↓
宿主 AppIntent → SDK 写 App Group（ControlWidget[intentName]）+ OpenURL 拉起主 App
  ↓
游戏主场景 ready 后 control/query → getInfo（读并消费磁盘标记）
  ↓
游戏按 intentName 分支，Native 内存保存 controlTask（产品层）
  ↓
游戏层调用服务端自动收花接口（带自生成 requestId 等）
  ↓
服务端判断有成熟鲜花并完成收花
```

Native 已实现：`query` 返回 `fromControlCenter`；`getInfo` 返回 `intentName` / `ts`。自动收花状态机见 [02-程序设计与接入说明](./02-程序设计与接入说明.md) §7。

### 5.2 无可收取鲜花

```text
用户点击系统控制按钮
  ↓
进入游戏并完成登录
  ↓
游戏层调用服务端自动收花接口
  ↓
服务端判断无成熟鲜花
  ↓
不执行收花，停留在游戏内
```

### 5.3 控制拉起后被打断

```text
用户点击系统控制按钮
  ↓
主 App 读取并清除 App Group pending request
  ↓
Native 内存保存 controlTask
  ↓
电话 / 闹钟 / 日历提醒 / 用户主动切后台
  ↓
如果尚未开始调用收花接口，则取消 controlTask
  ↓
用户之后手动回前台，不再自动收花
```

## 6. 边界条件

### 6.1 App Group 标志残留

若控制标记长期不消费，用户稍后手动打开 App 可能误触发业务。

处理策略（**已实现**）：

- App Group 使用 `ControlWidget` 目录，子键为 `intentName`，字段含 `ts`、`isControlOpen`。
- `query` / `getInfo` 命中有效项后 **删除对应子键**；过期（**30s**）项自动 purge。
- 游戏侧宜在 `getInfo` 后尽快转入内存态任务，勿依赖 App Group 长期存收花状态。

### 6.2 App 尚未启动就被打断

如果用户点击控制后，主 App 尚未获得任何生命周期回调就被打断，主 App 没机会读取并清除 App Group。

处理策略：

- pending request 必须有 TTL，建议 10 到 30 秒。
- 下次主 App 读取时，如果超过 TTL，直接清除并返回无效。

### 6.3 切后台或系统打断

如果在真正调用收花接口前发生 `sceneWillResignActive` 或 `sceneDidEnterBackground`，建议取消本次自动收花。

处理策略：

- `claimed`、`waitingForLogin`、`waitingForSceneReady` 状态可以取消。
- `executing` 状态不建议取消，因为请求可能已经发出，应交给服务端幂等处理。

### 6.4 服务端可信性

服务端无法绝对证明客户端上报的 `source=ios_control` 一定来自系统控制按钮，因为客户端字段理论上可能被 Hook 或篡改。

处理策略：

- 服务端必须校验登录态。
- 服务端必须使用 `requestId` 做幂等。
- 服务端必须重新判断是否存在成熟鲜花。
- 不应仅凭 `source=ios_control` 发放额外高价值奖励。

## 7. 验收标准

- iOS 18 设备可以添加控制按钮到控制中心、锁屏或操作按钮支持的位置。
- 点击控制按钮后可以拉起主 App（宿主 Intent + URL scheme）。
- `control/query` 在点击后短时内返回 `fromControlCenter=true`；再次 query 为 false（已消费）。
- `control/getInfo` 返回正确 `intentName`；超过 **30s** TTL 后 query 为 false。
- 控制拉起后，在收花接口执行前切后台或被电话、闹钟、日历提醒打断，本次自动收花取消。
- 已进入收花接口执行阶段时，客户端不重复触发，服务端通过 requestId 幂等处理。
- 无成熟鲜花时不执行收花，只正常进入游戏。
