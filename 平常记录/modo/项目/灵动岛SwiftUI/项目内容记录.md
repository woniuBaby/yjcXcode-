4. **不要在 Live Activity / SwiftUI 层直接使用这套复杂对象结构。**

> ✅ **推荐方案：**
>
> - **OC 层：完整解析服务端 JSON → 结构化 Model**
> - **传给 Live Activity 的 ContentState：只放“已扁平化、已计算好的展示数据”**

换句话说：

> **服务端 JSON 是“描述型数据”**
>  **Live Activity ContentState 必须是“结果型数据”**

### Live Activity 的硬性限制

`ContentState` 必须：

- `Codable`
- 尺寸极小（系统限制，常见问题）
- **不能包含复杂嵌套对象**
- **不能包含 Optional + 嵌套 + 多态结构**

5. Live Activity 的共享模型层 `ActivityAttributesModel`。

把 `ActivityAttributesModel` 拆到 `ActivityCoreDate`，让 `ActivitySDK` 和 `ActivityUI` 共用同一份模型，并避免灵动岛/Widget Extension 依赖带宿主能力的 `ActivitySDK`。

## 为什么单独一层

- **谁在用**：`ActivitySDK`（创建/更新/结束 Live Activity）、`ActivityUI`（灵动岛/锁屏/Widget 渲染）都要用，属于 ActivityKit 核心协议，既不是纯 UI 也不是纯业务，适合独立共享层。
- **之前**：两边各一份 → 改字段要改两处、默认值/解码可能不一致、版本易错位、排查难。
- **现在**：单一定义、单一默认值/解码、两边协议一致。

## 为什么不让 ActivityUI 直接依赖 ActivitySDK

- `ActivitySDK` 含宿主能力：UIKit、UIApplication、弹窗、跳转设置等，Extension 环境不能用/不该用。
- Extension 是系统托管进程，不能依赖 `UIApplication.shared`、`UIAlertController`、打开设置、前台生命周期等。
- ActivityUI 只需读模型、渲染 UI，不需要创建/结束 Activity、权限弹窗、跳转设置；为拿一个 model 引整包依赖过重。
- Extension 依赖“宿主能力 SDK”易导致：运行不安全、动/静态库冲突、CocoaPods 复杂、后续宿主改动误伤 Extension。

## 职责划分

| 模块 | 职责 |
|------|------|
| **ActivityCoreDate** | 共享模型（当前即 `ActivityAttributesModel`），无 UIKit、无宿主行为 |
| **ActivitySDK** | 创建/更新/结束 Live Activity，宿主侧能力 |
| **ActivityUI** | 灵动岛/锁屏/Widget 展示，只依赖 CoreDate，不依赖宿主能力 |

## 目标

单份模型、职责清晰、ActivityUI 对 Extension 轻且安全、改字段只改一处。
