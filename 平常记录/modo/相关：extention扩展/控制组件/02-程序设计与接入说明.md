# 控制组件：程序设计与接入说明

## 1. 分层（当前落地）

| 层 | 位置 | 职责 |
|----|------|------|
| Control Widget | 控制中心 Extension | 系统按钮 UI；绑**宿主** `AppIntent` |
| AppIntent（宿主双 Target） | 扩展 perform + 主 App wire | 调 SDK `HostedJump`：写盘 + `OpenURLIntent` |
| ControlWidgetCore | Pod | 写盘格式、URL、防 strip Bootstrap |
| NativeBility `PluginAdapter_control` | 主 App | `query` / `getInfo` / `delegate` |
| 游戏 / 服务端 | 业务 | 按 `intentName` 分支；自动收花、幂等等 |

实施步骤以 [05](./05-控制中心顺序链路.md)、[06](./06-最小接入与配置.md)、[07](./07-花园工程跳转主App与排障.md) 为准。下文 **§3～§6** 为**已实现**契约；**§7～§11** 为自动收花等**产品层建议**（游戏可自行实现，非插件强制字段）。

---

## 2. Target 与系统要求

- **iOS 18+**：`ControlWidget`、`AppIntents`。
- **推荐**：控制中心 **独立** Extension（`HYControlWidgetExtension`），Deployment **18.0**；桌面小组件用另一 Extension（见 [06](./06-最小接入与配置.md)）。
- 主 App 与扩展 **同一 App Group**；宿主 Swift（`HYControlWidgetOpenApp.swift`）勾选 **主 App + 扩展** 双 Target。

## 3. App Group 数据协议（已实现）

存储：`UserDefaults(suiteName:)`，suite 解析顺序：

1. 主 App / 扩展 plist `NTControlCenterAppGroupSuite`
2. `ControlWidgetCoreAppGroupSuite`
3. SDK / 插件推导 `group.{主包BundleId}`（扩展 bundle 会剥 `.HYControlWidgetExtension` 等后缀）

结构（与 `ControlWidgetCoreHostWire.h` 一致）：

```text
ControlWidget                    ← 根键 kNTControlWidgetFolderKey
  ├─ jump                        ← 子键 = intentName（业务 ID）
  │    └─ { ts, intentName, isControlOpen }
  └─ jumpWrite
       └─ { ts, intentName, isControlOpen }
```

| 字段 | 类型 | 说明 |
|------|------|------|
| `ts` | Number（Unix **秒**） | 写入时刻 |
| `intentName` | String | 与按钮业务 ID 一致 |
| `isControlOpen` | Bool | 固定 `true`（扩展写入） |

**写入方**：`ControlWidgetCoreControlCenterLaunchMarker.writeToControlWidgetFolder`，在 `HostedJump.perform` 中当 `definition.writeToGroup == true` 时调用。

**读取方**：`PluginAdapter_control`；校验 TTL、消费子键。

**不是**旧文档中的扁平键 `control.requestId` / `control.action`（若游戏需要 requestId，应在 **getInfo 之后** 由游戏层生成并带给服务端）。

### 3.1 TTL 与消费

| 参数 | 实现值 | 说明 |
|------|--------|------|
| `kMaxAge` | **30s** | 超过则无效并 purge |
| `kSkew` | **5s** | `ts` 在未来则无效 |
| 消费时机 | `query` / `getInfo` 命中有效项时 | 从 `ControlWidget` 删除对应子键 |

## 4. 扩展侧 AppIntent（宿主 + SDK）

示意（花园 `HYControlWidgetOpenJumpIntent`）：

```swift
public struct HYControlWidgetOpenJumpIntent: AppIntent {
  public static var title: LocalizedStringResource { "jump" }  // 必须字面量
  public static let openAppWhenRun: Bool = true

  @MainActor
  public func perform() async throws -> some IntentResult & OpensIntent {
    try await ControlWidgetCoreHostedJump.perform(
      definition: HYGardenControlDefinitions.Jump.definition
    )
  }
}
```

`ControlWidgetCoreHostedJump.perform`：

1. 若 `writeToGroup` → 写 `ControlWidget[intentName]`
2. `return OpenURLIntent(url: definition.openURL)`

`ControlWidget` 注册（仅扩展 Bundle）：

```swift
ControlWidgetButton(action: HYControlWidgetOpenJumpIntent()) { ... }
```

**禁止**作为唯一方案：`ControlWidgetCoreOpenAppControl` + `ControlWidgetCoreButtonIntent`（真机常见不跳 App）。

主 App **wire**（防 Release 裁剪）：

```swift
HYControlWidgetMainAppWire.wire()  // _ = HYControlWidgetOpenJumpIntent.self ...
ControlWidgetCoreWireHostAppIntentsIfNeeded()  // Bootstrap +load 也会调
```

## 5. 主 App 读取时机（游戏）

插件不强制在 `didFinishLaunching` 自动读盘；推荐游戏在 **主场景 ready** 后：

```text
control/default/query  （参数 {} 或带 intentName 过滤）
  ↓ fromControlCenter == true
control/default/getInfo
  ↓ 得到 intentName / ts
按 intentName 分支（jump / jumpWrite / 自扩展）
```

`query` 已从磁盘 **消费** 时，`getInfo` 优先用内存缓存（同一次启动）。

调试台 Mock：`query` / `delegate` 用 `{}`；`getInfo` 可带 `{ intentName: "jumpWrite" }` 测过滤。

日志：Xcode 过滤 **`[control]`**，会打印 suite、folder 子键、命中/过期原因。

## 6. NativeBility 插件 API（已实现）

插件名：`control`，适配器 `default`。

| API | 入参 | 成功返回 |
|-----|------|----------|
| `query` | 可选 `data.intentName` 过滤 | `{ fromControlCenter: true/false, matchedIntentName: "" }` |
| `getInfo` | 可选 `intentName` | `{ intentName, ts, isControlOpen }` 或 `{}` |
| `delegate` | 可选 `intentName`；缺省清空整个 `ControlWidget` | `{ ok: true }` |

未链入 `ControlWidgetCoreHostWire.h` 时编译 `NT_CONTROL_WIDGET=0`，`query` 恒 `fromControlCenter: false`。

JS 调用形态（与项目 nt 命名一致）：

```text
control/default/query
control/default/getInfo
control/default/delegate
```

## 7. 游戏层状态机（产品建议：自动收花）

若产品要在控制入口触发自动收花，建议在 **Native 内存** 维护任务态（非 App Group 长期存储）：

| 状态 | 说明 | 可取消 |
|------|------|--------|
| `claimed` | 已从 `getInfo` 领取 intent | 是 |
| `waitingForLogin` | 等登录 | 是 |
| `waitingForSceneReady` | 等主场景 | 是 |
| `executing` | 已调收花接口 | 否（服务端幂等） |
| `cancelled` / `done` | 终态 | — |

`sceneWillResignActive` / `sceneDidEnterBackground`：在 `executing` 之前可取消，避免用户切后台后仍自动收花。

---

## 8. 服务端（产品建议）

自动收花接口宜带客户端生成的 `requestId`（**非** App Group 现字段）、`source=ios_control`，服务端校验登录态、成熟状态、requestId 幂等。`source` 不能当作强安全证明。

---

## 9. 与旧设计稿差异

| 旧文档 | 当前实现 |
|--------|----------|
| `control.requestId` 等扁平键 | `ControlWidget` 目录 + `intentName` 子键 |
| `valid` / `reason` / `action=collectFlower` | `fromControlCenter` + `getInfo.intentName` |
| `CollectFlowerControlIntent` 示例 | 宿主 `HYControlWidgetOpen*Intent` + SDK HostedJump |
| `supportedModes.foreground` | `openAppWhenRun` + `OpenURLIntent` |
