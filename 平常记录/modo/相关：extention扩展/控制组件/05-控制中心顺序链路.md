# 控制中心：顺序链路与零件说明（白话版）

> 适用：**花园（HuaYuann）** 及「扩展 + 主 App + App Group + 游戏读盘」方案。  
> 工程：`build/ios/proj/HYControlWidget/`；SDK：`HYWidget/ControlWidgetCore`。

---

## 1. 用户点一下，系统里按什么顺序走

```
用户在控制中心点「花园·jump」
    │
    ▼
① 扩展进程：宿主 AppIntent.perform → SDK HostedJump（写 App Group、OpenURLIntent）
    │
    ▼
② 系统：用 URL Scheme 拉起主 App
    │
    ▼
③ 主 App 冷启动或从后台唤起
    │
    ▼
④ 主 App 启动时 wire 已保留 Intent 元数据
    │
    ▼
⑤ 游戏：NativeBility control 插件 query / getInfo
    │
    ▼
⑥ 游戏：按 intentName（jump / jumpWrite / …）分支
```

---

## 2. 每个零件是什么（白话）

| 零件 | 花园工程对应 | 一句话 |
|------|--------------|--------|
| **Intent** | `HYControlWidgetOpenJumpIntent` | 点击后执行；内部调 SDK 写盘 + 打开 URL |
| **Control** | `HYControlWidgetJumpControl` | 控制中心按钮 UI；**必须绑宿主 Intent** |
| **Bundle 一行** | `HYControlWidgetBundle.body` | 向系统登记有哪些控件 |
| **wire** | `HYControlWidgetMainAppWire` | 主 App 占住 Intent 类型，防 Release 不跳 |
| **URL** | `HYControlWidgetConfig.openURL` + 主 App plist | 打开主 App 的 scheme |
| **App Group** | SDK 写盘 + `PluginAdapter_control` 读 | 跨进程传 intentName |
| **游戏分支** | Lua/JS | 按 `intentName` 进不同逻辑 |

### 2.1～2.7 补充说明

- **Intent**：跑在扩展进程；`perform` 推荐 `ControlWidgetCoreHostedJump.perform(definition:)`。
- **Control**：`StaticControlConfiguration` + `ControlWidgetButton(action: 宿主Intent)`。
- **Bundle**：仅扩展、`@main` 唯一；每按钮一行 `XXXControl()`。
- **wire**：`_ = Intent.self`，无业务；`HYControlWidgetOpenApp.swift` 双 Target。
- **URL**：`OpenURLIntent`；主 App 须注册 scheme。
- **App Group**：子键 = `intentName`；suite 用 `NTControlCenterAppGroupSuite` 或 SDK 推导。
- **游戏**：SDK 只负责传准 `intentName`。

## 3. 配置集中在哪里（花园）

| 内容 | 位置 |
|------|------|
| 业务 ID | `HYControlWidgetIntentName` |
| 打开 URL、列表前缀 | `HYControlWidgetConfig` |
| 主包 ID（kind 用） | `HYControlWidgetConfig.hostBundleIdentifier`（运行时推导） |
| 每按钮文案 / kind / 图标 | `HYGardenControlDefinitions` + `HYControlWidgetSymbols` |
| `AppIntent.title` | 各 Intent 内**字面量**，与 case 同名 |

## 4. 新增一个按钮（当前推荐步骤）

| 顺序 | 做什么 | 位置 |
|------|--------|------|
| ① | 业务 ID | `HYControlWidgetIntentName` 加 `case` |
| ② | 配置 | `HYGardenControlDefinitions` 加枚举 + `openAppDefinition`；`HYControlWidgetSymbols` 加图标 |
| ③ | 宿主 Intent | 复制 `HYControlWidgetOpenJumpIntent`；`title` 字面量 = rawValue；`perform` 用对应 `definition` |
| ④ | 宿主 Control | 复制 `HYControlWidgetJumpControl`，绑新 Intent |
| ⑤ | Bundle | `HYControlWidgetBundle` 加一行 `…Control()` |
| ⑥ | wire | `wire()` 加 `_ = 新Intent.self` |
| ⑦ | 游戏 | `intentName` 分支 |

**Target**：`HYControlWidgetOpenApp.swift` → 主 App + 扩展；`HYControlWidgetBundle.swift` → 仅扩展。

验收：删控件再加 → 冷启动 → `query` 用 `{}`。

## 5. 常见问题

### 5.1 是否都要双 Target？

只有 **控制中心 + 宿主 AppIntent** 这条链需要；通知、桌面小组件等规则不同。

### 5.2 SDK 怎么用？

- 主 App + 扩展都链 **ControlWidgetCore**。
- **perform / 写盘** 用 SDK；**按钮 action** 用 **宿主 Intent**。
- **勿** Bundle 仅 `ControlWidgetCoreOpenAppControl`（真机易 **完全不跳 App**）。

### 5.3 代码为什么仍要多？

系统模型决定：每入口仍要 **宿主 Intent + Control + Bundle 一行 + wire**；SDK 省的是 perform / 写盘重复代码。

## 6. 技术难点（为什么必须这样拆）

| 现象 | 根因 | 当前做法 |
|------|------|----------|
| 真机点了完全不进 App | `ControlWidgetButton` 绑 Pod 内 `ControlWidgetCoreButtonIntent`，主 App 对不齐 App Intents 元数据 | **宿主** `HYControlWidgetOpen*Intent` + 双 Target |
| Debug 跳、Release 不跳 | Intent 类型被链接器裁掉 | `MainAppWire` + `ControlWidgetCoreBootstrap` |
| `query` 永远 false | 扩展写了盘、主 App suite 不一致；或 `query` 带了错误 `intentName` | 对齐 `NTControlCenterAppGroupSuite`；调试 `{}` |
| 改代码控件行为不变 | 系统缓存控制中心配置 | 用户删除控件再添加 |
| 编译 `title` 报错 | Apple 要求 `AppIntent.title` 编译期常量 | 每个 Intent 写 `"jump"` 等字面量 |
| 能进 App 但游戏无分支 | 未调 `getInfo`；或 `jump` 按钮 `writeToGroup=false` | 确认 `writeToGroup`；ready 后 query→getInfo |

SDK 只统一 **写盘 + OpenURL**；**按钮 action 类型** 必须落在宿主，这是系统模型限制，不是 Pod 漏功能。
